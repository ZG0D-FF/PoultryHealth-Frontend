Chapter 03

⚡ The 1-Minute Summary

If you read nothing else, read this.

Microphone listens

A USB mic on a Raspberry Pi records audio continuously at 44.1kHz. When it detects a sound louder than threshold (0.15 amplitude), it starts recording until silence returns.

Audio is cleaned

The recording is downsampled to 16kHz and passed through a bandpass filter (300–4000 Hz) to isolate chicken vocalization frequencies and remove farm machinery noise.

YAMNet checks: "Is this a bird?"

Google's pre-trained YAMNet model (521 audio classes) checks if the sound is bird-related. If the bird score is below 0.05, it's classified as background noise and ignored.

80 features are extracted

The audio is analyzed for 80 bioacoustic features: MFCCs, pitch, energy, spectral shape, rhythm, and more. These numbers become the "fingerprint" of the sound.

AI models diagnose

A cascade of models (SVM → Logistic Regression → HoneyBee Queen stacker) votes on whether the chicken is Healthy or Sick. A human can confirm or override via terminal.

Alert + Archive

If sick: Telegram alert sent. The audio is saved to a "correction vault" with its label for future retraining. The web dashboard updates in real-time.

Chapter 05

🔄 The Audio Pipeline

What happens to sound from microphone to diagnosis.

🎤 Step 1: Recording

The Pi records 0.5-second chunks at 44,100 Hz. When a chunk's peak amplitude exceeds 0.15, the system starts capturing. It keeps recording until 3 consecutive chunks are quiet (below 0.10), or 15 seconds max.

🧹 Step 2: Preprocessing

Audio is resampled from 44,100 Hz → 16,000 Hz, then cleaned with a 5th-order Butterworth SOS bandpass filter. This keeps only the 300–4,000 Hz range — where chicken vocalizations live. Farm machinery, wind, and low rumbles are removed.

🚪 Step 3: YAMNet Gatekeeper

Google's pre-trained yamnet.tflite model (521 audio classes) evaluates the sound. It sums the probabilities of bird-related classes (indices 73-81, 93, 100-102). If the total is below 0.05, the sound is ignored as "not a bird." This prevents the system from misdiagnosing a tractor engine as a sick chicken.

🔬 Step 4: Feature Extraction

80 numerical features are computed from the audio using librosa. These capture everything: pitch, rhythm, energy, spectral shape, noisiness. See the 80 Features section for the full breakdown.

🛡️ Step 5: Z-Score Anomaly Shield

The 80 features are normalized using a StandardScaler (fitted during training). If any normalized feature has an absolute value greater than 6.0, the sound is classified as "Alien Noise" — it's so far from any chicken the system has ever heard that it can't possibly be one.

⚔️ Step 6: Cascade Inference

See The Cascade section for full details.

🔬 Level 5 — Byte & Signal Level

What: What happens to individual audio samples and bytes.

🎤

Microphone → ADC

USB mic ADC samples air pressure 44,100×/sec. Each sample = 1 float32 = 4 bytes = 32 bits (1 sign + 8 exp + 23 mantissa). Range: -1.0 to +1.0.

📉

Resampling

Polyphase FIR anti-aliasing filter + decimation factor 2.756×. 132,300 samples → 48,000. Frequencies above 8kHz are removed (Nyquist).

🧹

Bandpass (SOS)

5th-order Butterworth, 3 biquad sections. Forward+backward (zero-phase). y[n] = b0·x[n] + b1·x[n-1] + b2·x[n-2] - a1·y[n-1] - a2·y[n-2]. ~54 FLOPs/sample.

📐

STFT → Spectrogram

2048-sample window (128ms). Hop=512 (32ms). FFT → 1025 complex bins/frame. Each complex = 16 bytes. Output: ~1.47 MB spectrogram.

🎵

Mel + MFCC

128 mel-scale triangular filters → log dB → DCT → 13 cepstral coefficients/frame. Mean across time → 13 floats = 104 bytes. The acoustic fingerprint.

🎯

Feature Vector

80 features × 8 bytes = 640 bytes total. That's the entire chicken sound compressed from 192 KB audio into 640 bytes. A 300× compression ratio.

⚖️

StandardScaler

For each feature: (x - μ) / σ. Mean and std from training. 2 FLOPs per feature × 80 = 160 FLOPs. Still 640 bytes but now zero-mean, unit-variance.

🗳️

SVM Decision

K(x,sv) = exp(-γ||x-sv||²) for ~200 support vectors. ~32,000 FLOPs. Output: 2 probs = 16 bytes. argmax → 1 bit: Healthy (0) or Sick (1).

The journey of one chicken sound:
🎤 Air vibrations → 88,200 bytes (0.5s raw) → resampled to ~32,000 bytes → bandpass → STFT to ~1.5 MB spectrogram → collapsed to 640 bytes (80 features) → normalized → SVM kernels → 16 bytes (2 probs) → 1 bit: Healthy or Sick.
Chapter 09

🔌 Hardware & Architecture Evolution

Why the Raspberry Pi, and why shift to the ESP32?

🍓 Why use Raspberry Pi 3B and a USB Mic?

Rapid Prototyping: The Pi 3B runs a full Linux OS (Debian/Ubuntu), which makes it incredibly easy to run standard Python libraries like librosa, scikit-learn, and sounddevice without complex cross-compilation.

USB Microphone: A plug-and-play USB microphone has an integrated ADC (Analog-to-Digital Converter) and bypasses the noise typically found on a Pi's analog audio inputs. However, USB mics are bulky and draw significant power.

⚡ Why shift to ESP32 with 4+ I2S Mics?

Power & Cost: A Raspberry Pi requires ~3-5 Watts and costs ~$35+. An ESP32 microcontroller uses <0.5 Watts and costs ~$3. It can run on batteries or small solar panels.

I2S Microphones (e.g., INMP441): I2S is a digital protocol designed specifically for audio. Using 4 or more I2S mics allows for Spatial Audio Processing. By calculating the Time Difference of Arrival (TDOA) between the mics, the AI can triangulate exactly which chicken is sick in a large barn.

Real Edge Deployment: The ESP32 eliminates the overhead of an OS. It boots instantly and handles real-time interrupts perfectly.

🧠 Why TinyML?

TinyML brings machine learning directly to microcontrollers. Instead of sending 44.1kHz audio streams to the cloud (which takes massive bandwidth, incurs latency, and raises privacy issues), TinyML allows the ESP32 to compute the FFT, extract MFCCs, and run an SVM directly on its 240MHz processor. It only sends a few bytes (the "Sick" alert) over the network.

Chapter 10

🧮 The Mathematics of SmartEars

Explaining the core algorithms in deep mathematical detail.

🌊 Why FFT (Fast Fourier Transform)?

Sound is recorded as changes in air pressure over time (Time Domain). But chicken diseases (like rattling/wheezing) manifest as specific frequency bands. The FFT converts Time Domain → Frequency Domain.

The Math: For a discrete signal x[n], the DFT (Discrete Fourier Transform) is:
X[k] = Σ (n=0 to N-1) x[n] · e^(-i·2π·k·n/N)

This complex exponential breaks the sound down into constituent sine waves (Euler's formula). The FFT is an O(N log N) algorithm to compute this rapidly.

🎵 Mel-Frequency Cepstral Coefficients (MFCC)

Why MFCC instead of raw FFT? Because human (and animal) hearing is not linear. We easily tell the difference between 100Hz and 200Hz, but not between 10,000Hz and 10,100Hz.

  1. Mel Filterbank: We apply overlapping triangular filters to the power spectrum. The spacing is linear below 1kHz and logarithmic above. m = 2595 · log10(1 + f/700)
  2. Logarithm: We take the log of the filter energies because loudness perception is logarithmic (decibels).
  3. DCT (Discrete Cosine Transform): We apply a DCT to the log-mel energies to decorrelate the features. This collapses the spectral envelope into just 13 numbers.

🤖 Why SVM and RF over Deep Learning?

Deep Learning (like CNNs/ResNets) requires massive datasets (millions of samples) and heavy compute power (GPUs). SmartEars uses classical Machine Learning for three reasons:

  1. Data Efficiency: SVMs and Random Forests can train highly accurate boundaries with only hundreds or thousands of labeled samples.
  2. Feature Engineering: Because we mathematically extract 80 highly targeted bioacoustic features (using DSP), the models don't have to "learn how to listen." They just learn the boundaries.
  3. Edge Computing: An SVM prediction (RBF kernel) requires computing K(x, x_i) = exp(-γ||x - x_i||^2) for the support vectors. This requires only a few thousand floating-point operations, easily running on an ESP32 in microseconds.

💬 FAQ

Questions you'll probably have.

Where do I retrain the models?

On Google Colab using v12.py. Upload your audio data to Google Drive under /MyDrive/CHICKENAI/ with Healthy/ and Sick/ subfolders. Run all cells. Download the .pkl files and transfer to the Pi.

What hardware is needed?

Raspberry Pi 3B (or newer) + any USB microphone. The Pi needs Python 3.7+ with librosa, scikit-learn, sounddevice, websockets installed. No GPU needed — inference is CPU-only.

Can it work without internet?

Mostly yes. Core detection works fully offline. You lose: Telegram alerts, J.A.R.V.I.S. chat (Groq API), and the browser dashboard if you're remote. The dashboard works on the local network.

What's the "correction vault"?

Every audio clip the system analyzes is saved with its label (healthy/sick/noise) in correction_vault/. When a human overrides the AI's diagnosis, the corrected label is used. These recordings can be fed back into training to improve the models.

What's the accuracy?

V11 ensemble achieved 75% accuracy. V12 with the HoneyBee Queen meta-learner achieved ~60% accuracy on real-world noisy data. The confusion matrices show ~60% healthy recall and ~60% sick recall.

What do the dashboard passwords mean?

Both index.html and poultryhealth_v12_titan.html have a lock screen. Accepted passwords: cluck, cluck!, CLUCK, bawk, cluckcluck. It's a fun chicken-themed gate.

What's the difference between index.html and the V12 HTML?

index.html is the V11 green cyberpunk theme ("HONEY BEE HIVE"). poultryhealth_v12_titan.html is the V12 deep blue "MONARCH ORCA" theme with specimen batch IDs and "ANOMALY" labeling. Same WebSocket connectivity, different aesthetics.

Chapter 15

📚 Deep Theoretical Foundations

The underlying physics, mathematics, and edge logic powering SmartEars.

1. IoT Edge Architecture & Constraints

The IoT Pipeline Hierarchy: Sending raw audio directly to the Cloud is a rookie mistake that creates severe lag and drains battery. This reinforces why SmartEars must process the FFT and SVM locally at the Edge, passing only lightweight actionable intelligence up the chain.

Hardware Interrupts vs. Polling: Our monitor.py avoids "busy-waiting" (polling) which would freeze the CPU entirely. Instead, it uses event-driven interrupts to handle incoming audio chunks seamlessly, preventing the system from bottlenecking.

Microcontroller Memory Loadouts: Our .pkl SVM weights and TFLite models are stored securely in Non-Volatile Flash memory, while the runtime ORCA_SENSORY_DATA is processed rapidly in the tiny, volatile SRAM.

2. Audio Data Acquisition Mathematics

Sampling vs. Quantization: Capturing physical sound requires splitting it in two ways: discretizing time (Sample Rate, like 16kHz) and discretizing amplitude (Quantization).

Signal-to-Quantization-Noise Ratio (SQNR): Dictated by the formula SQNR = 6.02N + 1.76 dB. This explains exactly why we utilize high-bit digital I2S microphones. Every single bit added to the ADC hardware crushes the quantization noise floor by exactly 6 dB, pulling the faint respiratory sounds of a sick chicken cleanly out of the background fuzz.

3. Agentic AI (Explaining J.A.R.V.I.S.)

Tools and Function Calling: J.A.R.V.I.S. (Groq) is not just a passive chatbot predicting the next token. It operates as an "Agentic AI" that has been given access to Python functions (Tools), allowing it to halt text generation, trigger real-world actions (sending Telegram alerts), and evaluate physical states.

Safety Drift & Hallucinations: To prevent the LLM's safety protocols from slowly degrading or hallucinating fake disease states over a long continuous stream, our system relies on strict, structured prompt bounds and a hard 10-second confirmation window.

4. Advanced Anomaly Detection (The Future of the Shield)

Autoencoders / VAEs: Currently, we use a rigid math-based Z-Score Shield (Z > 6.0 = Alien Noise). The future evolution of this shield is an Autoencoder used for density estimation. By learning the exact latent probability distribution of a "normal" barn environment, the AI will instantly flag any outlier sound (like a tractor engine) simply because it falls outside the mathematically learned distribution.

5. I²S (Inter-IC Sound) Protocol

Raw PCM Audio Streaming: Unlike I²C, I²S is specifically optimized for streaming PCM audio data. Our digital microphone streams raw audio to the edge device using this protocol. By strictly separating the clock (SCK), word select (WS for L/R channels), and data (SD) lines, I²S minimizes physical jitter. This is absolutely critical for ensuring mathematically clean FFT conversions before the AI even touches the data.

6. DMA (Direct Memory Access) Transfers

Zero-Latency Concurrency: Our system bypasses the CPU for high-speed audio operations. The DMA controller funnels the incoming I²S audio stream directly into the ESP32/Pi's SRAM. Because the CPU is completely bypassed during data transfer, it can continuously crunch the heavy SVM math without ever dropping or missing incoming audio frames.

7. Hardware Logic Level Shifting

Voltage Constraints: A core physical engineering constraint at the edge. We utilize logic level shifters to step down voltages (e.g., from 5V sensor logic to 3.3V Pi logic) to safely communicate between different hardware nodes without frying the sensitive receiver pins.

8. UART Hardware FIFOs & Flow Control

Preventing Data Overrun: We rely on UART First-In-First-Out (FIFO) hardware buffers. When the edge CPU is locked up at 100% capacity running the HoneyBee Queen cascade inference, these hardware buffers physically hold the incoming bytes, ensuring no physical sensor telemetry is lost while the processor is busy.

Chapter 16

Microcontrollers & Physical Layers

1. I²S Audio Acquisition (The SmartEars Core)

PCM & The Nyquist Theorem: To accurately capture the faint respiratory sounds of a sick chicken without aliasing distortion, the analog audio signal must be sampled at a rate at least twice the highest frequency (Nyquist Theorem). We utilize the I²S protocol, which is optimized exclusively for streaming this PCM (Pulse Code Modulation) audio data.

Quantization & Bit Depth: Digital microphones convert analog amplitudes into discrete values. A higher bit depth (e.g., 16-bit or 24-bit) drastically lowers the quantization noise floor, pulling clean audio out of the background fuzz.

I²S Pins: The protocol strictly separates the clock and data lines to minimize physical jitter: Serial Data (SD) for the audio bits, Word Select (WS) to toggle left/right channels, and Serial Clock (SCK) for synchronization.

2. Microcontroller Serial Protocols

UART (Telemetry): An asynchronous protocol ideal for lightweight text alerts (like sending SICK/HEALTHY to a WiFi module). It relies on Start, Stop, and Parity bits rather than a shared clock. Baud Rate (signal changes per second) dictates the speed, which both devices must agree upon beforehand.

SPI (High-Speed Data): A synchronous, full-duplex protocol designed for high-speed peripherals (like SD cards or fast sensors). It uses Master Out Slave In (MOSI), Master In Slave Out (MISO), Serial Clock (SCK), and Slave Select (SS). It supports multiple Clock Polarity (CPOL) and Phase (CPHA) modes, and devices can even be daisy-chained.

I²C (Sensor Expansion): A synchronous, half-duplex protocol using only two wires (SDA and SCL). It uses a Master/Slave architecture with 7-bit addresses, Start/Stop conditions, and ACK/NACK bits. It relies on a command-response cycle, making it perfect for wiring up additional environmental sensors (like the Senseair K30 CO2 sensor) to the same bus without consuming extra pins.

3. The Edge ML Ecosystem

TensorFlow Lite for Microcontrollers: A drastically optimized version of the TF Lite framework designed to run neural networks on embedded devices with only tens of kilobytes of memory.

ESP-IDF Environment: The official development framework for Espressif chips. We compile and flash our C/C++ firmware to the ESP32 edge device using the ESP-IDF toolchain to achieve maximum bare-metal performance.
